혜성이 온다... 아, 죄송합니다. 짧지만 깨달음을 주는 주석입니다. 그렇다면 주석은 정확히 무엇이고, 왜 필요한 걸까요? 사실, 컴파일러가 무시하는 특수 텍스트라고 할 수 있습니다. 주석을 사용하면 코드의 특정 부분을 명확하게 설명하거나 컴파일 과정에서 제외(즉, 비활성화)할 수 있습니다. 이 강좌에서는 주석을 사용하여 코드의 작동 방식과 이유를 설명하겠습니다.

코틀린에는 세 가지 종류의 주석이 있습니다. 하나씩 살펴보겠습니다.

## 줄 끝 주석

"//" 다음에 나오는 모든 텍스트는 컴파일러가 무시합니다.

```kotlin
fun main() {
    // The line below will be ignored
    // println("Hello, World")

    // This prints the string "Hello, Kotlin"
    println("Hello, Kotlin")  // Here can be any comment
    /// Valid single-line comment
}
```

**알고 계셨나요?** 대부분의 코드 편집기에서는 캐럿을 해당 줄에 놓고 `Ctrl /`또는 를 눌러서 줄에 주석을 달 수 있습니다 `cmd /`.

## 여러 줄 주석

컴파일러는 < `/*`와 `*/`< 사이의 모든 텍스트를 무시합니다. 이 기호들을 사용하여 한 줄 주석과 여러 줄 주석을 모두 작성할 수 있습니다.

```kotlin
fun main() {
    /* This is a single-line comment */
    /*  This is an example of
        a multi-line comment */

    /*** Valid multiline comment
        println("Hello")
        println("World")
    **/
}
```

다른 주석 안에 주석을 넣을 수도 있습니다. 중첩된 [여러 줄 주석을](https://hyperskill.org/learn/step/4406 "코틀린에서는 여러 줄 주석(문서 주석 또는 doc 주석이라고도 함)을 사용하여 코드를 문서화하고 API 문서를 생성합니다. | 이러한 주석은 `/**`로 시작하고 `*/`로 끝나는 특정 형식으로 작성되며, 서식 지정을 위해 마크다운을 포함할 수 있습니다. 문서화해야 하는 클래스, 메서드 또는 함수의 선언 바로 위에 작성합니다. doc 주석의 목적은 코드에 대한 자세한 설명을 제공하여 다른 사람들이 해당 기능을 더 쉽게 이해할 수 있도록 하는 것입니다.")`/*` 작성할 때는 시작 기호 와 종료 기호가 쌍을 이루도록 하세요 `*/`.

```kotlin
fun main() {
    /*
    System.out.println("Hello")  // print "Hello"
    System.out.println("Kotlin") /* print "Kotlin" */
    */
}
```

## 문서 주석(doc comments)

컴파일러는 여러 줄 주석의 모든 텍스트를 무시하는 것처럼 기호 `/**`와 기호 사이의 모든 텍스트를 무시합니다. 이러한 종류의 주석을 **문서 주석** ( **doc comments** )이라고 합니다. 문서는 다른 사람들이 코드의 기능을 이해하는 데 도움이 되는 텍스트입니다. 이러한 종류의 주석을 사용하면 특수 도구를 사용하여 소스 코드에 대한 [문서를 자동으로 생성할](https://hyperskill.org/learn/step/4406 "In Kotlin, generate documentation refers to the process of creating documentation for your code using tools such as Dokka. | This documentation is generated from comments written in a specific syntax called Kdoc, which allows developers to write comprehensive descriptions and explanations for their code. Kdoc supports various block tags such as `@param` for documenting parameters, `@return` for documenting return values, `@constructor` for documenting constructors, and many more. These comments, written in Markdown, are used to generate HTML documentation that can be used by other developers to understand the code. Dokka is a documentation generator for Kotlin, similar to Javadoc for Java, that generates API documentation from comments in the source code.") 수 있습니다 .`*/`[](https://hyperskill.org/learn/step/4406 "Kotlin에서 문서 생성은 Dokka와 같은 도구를 사용하여 코드에 대한 문서를 만드는 과정을 의미합니다. | 이 문서는 Kdoc이라는 특정 구문으로 작성된 주석에서 생성되며, 개발자는 이를 통해 코드에 대한 포괄적인 설명과 해석을 작성할 수 있습니다. Kdoc은 매개변수를 문서화하는 `@param`, 반환 값을 문서화하는 `@return`, 생성자를 문서화하는 `@constructor` 등 다양한 블록 태그를 지원합니다. 이러한 주석은 Markdown으로 작성되며, 다른 개발자가 코드를 이해하는 데 사용할 수 있는 HTML 문서를 생성하는 데 사용됩니다. Dokka는 Java의 Javadoc과 유사한 Kotlin용 문서 생성기로, 소스 코드의 주석에서 API 문서를 생성합니다.")

일반적으로 이러한 주석은 각 프로그램 요소의 선언 위에 배치됩니다. 또한, 이 경우 도구 제어를 위해 , , 등과 같은 특수 레이블이 사용됩니다 `@param`.`@return`

아래의 예를 살펴보겠습니다.

```kotlin
/**
 * The `main` function accepts string arguments from outside.
 *
 * @param args arguments from the command line.
 */
fun main(args: Array) {
    // do something
}
```

**아직 문서 주석을** 완전히 이해하지 못하더라도 걱정하지 마세요 . 나중에 더 자세히 살펴보겠습니다.

## 결론

코드에 삽입할 수 있는 여러 가지 주석 유형에 대해 간략하게 살펴보았습니다. 주석을 현명하게 사용하고, 주석이 많다고 해서 프로그램이 항상 완벽하게 명확해지는 것은 아니라는 점을 명심하십시오. 코드가 변경될 수 있으며, 주석은 시대에 뒤떨어질 수 있습니다. 주석의 수를 줄이려면 자체 문서화 코드를 작성해야 하는데, 초보자에게는 쉽지 않을 수 있습니다. 이 과정에서 그 방법을 배우게 될 것입니다.
